# Documentation Conventions

This document defines how Kiki OS documentation is authored, named, structured, and validated. It governs the docs themselves; it is not a SPEC for any system component.

## Frontmatter

Every document begins with YAML frontmatter:

```yaml
---
id: <kebab-case-stable-id>
title: <Human Title>
type: DESIGN | SPEC | GUIDE | ADR | RFC
status: draft | stable | deprecated | superseded-by:<id>
version: <semver>
implements: [<feature-id>, ...]
depends_on: [<doc-id>, ...]
depended_on_by: [<doc-id>, ...]
research_sources: [<url>, ...]
last_updated: <ISO-date>
---
```

- `id` is stable forever. Renames break references.
- `version` tracks the document, not the underlying feature.
- `depends_on` is authored manually. `depended_on_by` is auto-generated by the doc linter.
- `research_sources` is required where a doc cites external research, optional otherwise.

## Naming

- IDs in frontmatter: `kebab-case`, stable.
- Filenames: `UPPERCASE-WITH-DASHES.md`.
- Paths always relative: `02-platform/SANDBOX.md`. Never reference by name only.

## Document structure by type

### DESIGN

```
# Title
## Problem
## Constraints
## Decision
## Rationale
## Consequences
## References
```

### SPEC

```
# Title
## Purpose
## Inputs
## Outputs
## Behavior
   (subsections per concern)
## Interfaces
   (programmatic API, CLI, config)
## State
   (persistent, in-memory)
## Failure modes
   (table)
## Performance contracts
## Acceptance criteria
   (checklist)
## Open questions
## References
```

### ADR

```
# ADR-NNNN: Title
## Status
## Context
## Decision
## Consequences
## Alternatives considered
## References
```

### RFC

```
# RFC-NNNN: Title
## Summary
## Motivation
## Detailed design
## Drawbacks
## Alternatives
## Unresolved questions
## Adoption strategy
```

### GUIDE

Free-form, but typically:

```
# Title
## Purpose
## How to do X
## Examples
## References
```

## Code samples

- Pseudocode (Rust-flavored) is permitted for algorithm descriptions.
- TOML/YAML for configurations and manifests.
- ASCII art for portable diagrams.
- Mermaid only when ASCII does not scale.
- Containerfile snippets for app packaging examples.

## Status flow

```
draft → review → stable → eventual deprecated (with superseded-by:<id>)
```

Only `stable` documents may be load-bearing references for other `stable` documents. `draft` documents may reference each other freely.

## Versioning

- Major version bump: requires an RFC.
- Minor version bump: usually requires an ADR.
- Patch version bump: clarifications only.

## Cross-references

Always use relative paths: `02-platform/SANDBOX.md`. Never use names alone (ambiguous when files are renamed across chapters).

## Authoring style

- Prefer short, declarative sentences.
- Define vocabulary in `00-foundations/GLOSSARY.md`. Avoid redefining elsewhere.
- State decisions explicitly. Avoid hedging when the decision is made.
- Tables and bulleted lists are preferred over prose for enumerations.
- Code examples should be minimal. Full implementations live in code, not docs.

## Validation

A linter validates:

- Frontmatter parses and contains required fields.
- All `depends_on` references resolve to existing documents.
- All cross-reference links resolve.
- No `stable` document references a `draft` document as load-bearing.
- IDs are unique across the corpus.
- File names match `id` in frontmatter.

The linter runs in CI; PRs that break it cannot be merged.

## Auto-generated meta

The following live in `meta/` and are regenerated on every CI run:

- `meta/DEPENDENCY-GRAPH.md` — full dependency graph derived from frontmatter.
- `meta/COVERAGE-MATRIX.md` — acceptance criteria mapped to implementing SPECs.

Do not edit `meta/` by hand.
