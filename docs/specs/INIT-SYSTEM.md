# Init System

## Purpose

Specify systemd as the init and supervision system for Kiki OS, the conventions for unit definitions, the user/system distinction, and the integration points with bootc, sysext, podman quadlet, and the agent harness.

## Inputs

- The base OS image with systemd 257+ installed.
- Unit files from the Kiki Runtime sysext.
- User session units (cage + agentui).
- App quadlet units (generated by `agentctl install`).
- systemd configuration in `/etc/systemd/`.

## Outputs

- A booted system with all required services running.
- Per-service supervision (restart on failure, watchdog, journald).
- Ready signaling to bootc and downstream tools.

## Behavior

### Why systemd

- It is the de facto standard for modern Linux distributions.
- Native integration with bootc, sysext, sysupdate, cryptenroll, homed, networkd/NetworkManager, journald.
- Provides Landlock unit directives, RestrictNamespaces, sandbox primitives at the unit level.
- Quadlet integrates podman containers as systemd units.
- Soft-reboot, kexec, generators, timers — all primitives we use.
- Ecosystem-wide assumption: most upstream Linux software ships systemd units.

Going against systemd would mean reimplementing or replacing much of this. We accept systemd as a substrate.

### Unit categorization

```
/usr/lib/systemd/system/             system units shipped in the OS image
/etc/systemd/system/                 system overrides (rare; configuration-controlled)
/usr/lib/systemd/user/               user units shipped in the OS image
/etc/systemd/user/                   user overrides

/run/systemd/system/                 transient units (e.g., per-call ephemeral tools)

/etc/containers/systemd/             quadlet definitions for app containers
                                     (generated by agentctl from app Profiles)
```

### Kiki targets

```
kiki-base.target              brings up base system services
   ├── systemd-cryptenroll    unseal LUKS keys with TPM
   ├── /var, /home mounts     after decryption
   ├── journald               structured logging
   ├── NetworkManager         network connectivity
   ├── systemd-resolved       DNS
   ├── pipewire system        audio service
   ├── kiki-hald-*            HAL daemons
   └── systemd-sysext         mount Kiki Runtime sysext

kiki-runtime.target           brings up the agent harness
   ├── memoryd
   ├── inferenced
   ├── toolregistry
   ├── policyd
   └── agentd
   (in dependency order; agentd last)

kiki-shell.target             user session
   └── cage (which launches agentui)
```

The default boot target is `graphical.target`, which depends on `kiki-shell.target`, which depends on `kiki-runtime.target`, which depends on `kiki-base.target`.

### Unit conventions

All Kiki-specific units follow conventions:

```ini
[Unit]
Description=Kiki: <component name>
Documentation=kiki:core/<component>
After=kiki-base.target
PartOf=kiki-runtime.target

[Service]
Type=notify
User=kiki-runtime
Group=kiki-runtime
ExecStart=/usr/bin/<binary>
Restart=on-failure
RestartSec=2
WatchdogSec=30
TimeoutStartSec=30
TimeoutStopSec=10
StartLimitIntervalSec=300
StartLimitBurst=5

# Sandboxing directives (Landlock, seccomp, namespaces)
ProtectSystem=strict
ProtectHome=read-only
PrivateTmp=true
PrivateDevices=true
NoNewPrivileges=true
RestrictNamespaces=~CLONE_NEWUSER
LockPersonality=true
RestrictRealtime=true
RestrictSUIDSGID=true
LandlockReadOnly=/usr /etc
LandlockReadWrite=/var/lib/kiki/<component>

[Install]
WantedBy=kiki-runtime.target
```

The exact directives vary per service. The conventions ensure every Kiki service has documented sandboxing.

### Notify protocol

Daemons signal readiness via `sd_notify(READY=1)`. systemd waits for the notification before declaring the unit started. Watchdog uses `sd_notify(WATCHDOG=1)` periodically; missing pings trigger restart.

### Quadlet for apps

Apps are not directly systemd-defined services. They are containers run by podman. Quadlet generates systemd units from `.container` files:

```ini
# /etc/containers/systemd/kiki-acme-notes--<workspace_id>.container
[Container]
Image=registry.acme.dev/notes:1.2.3
Volume=/var/lib/kiki/apps/acme/notes/data-<workspace_id>:/data:rw
Network=app-acme-notes
Environment=KIKI_WORKSPACE=<workspace_id>

[Service]
Restart=on-failure
Type=notify

[Install]
WantedBy=kiki-shell.target
```

Quadlet generates `kiki-acme-notes--<workspace_id>.service` from this. systemd treats it like any other unit. `agentctl install` creates the `.container` file; `agentctl uninstall` removes it.

### User session

The user's graphical session runs as systemd-user units, scoped to the user's uid:

```
~/.config/systemd/user/kiki-shell.target
   └── cage.service (user-scoped)
```

cage launches agentui as its single Wayland client. systemd-user supervises both.

### Hot reload

`systemctl daemon-reload` re-reads unit files after they change. `systemctl restart <unit>` restarts a service. The Kiki update orchestrator uses these primitives directly.

### Boot order summary

```
1. systemd PID 1 starts.
2. systemd-cryptenroll unseals disk keys via TPM PCRs 7,11,12,15.
3. /var, /home mount with decrypted keys.
4. journald starts.
5. kiki-base.target dependencies bring up network, audio, HAL, sysext.
6. kiki-runtime.target starts the 5 daemons.
7. systemd-user starts the user session.
8. cage launches; agentui starts as cage's client.
9. agentd discovers apps, registers tools.
10. system ready (boot to first interaction <30s on reference hardware).
```

### Soft-reboot

systemd's `systemctl soft-reboot` re-pivots userspace to a new rootfs without kernel reboot. Kiki uses this for sysext refresh that affects core daemons; the kernel stays running, only userspace transitions to the new state.

### kexec

For full kernel/base updates, `bootc upgrade` switches deployments and triggers a kexec or full reboot. systemd handles the orchestration.

## Interfaces

### Programmatic

The agent harness uses systemd's DBus API (via zbus) to:

- Start/stop/restart services.
- Subscribe to unit state changes.
- Generate transient units for per-call tool dispatch.

### CLI

Standard systemd:

```
systemctl status <unit>
systemctl restart <unit>
journalctl -u <unit>
```

Plus Kiki-specific:

```
agentctl status                  # Kiki Runtime overview
agentctl restart-runtime         # systemctl restart kiki-runtime.target
```

## State

systemd state in /run/systemd/. Persistent unit configuration in /usr/lib/systemd/ (sealed in image) and /etc/systemd/ (configuration-controlled).

## Failure modes

| Failure | Response |
|---|---|
| Service crashes | systemd restarts per Restart= directive |
| Service exceeds StartLimitBurst | systemd refuses to restart; alert |
| Watchdog timeout | systemd kills with SIGABRT; restarts |
| Dependency cycle | detected at unit-load time; aborts boot |
| Unit file syntax error | unit fails to load; alert; rest of system continues |

## Performance contracts

- systemd PID 1 startup: <500ms.
- All system services ready (kiki-base.target reached): <8s on reference hardware.
- All runtime services ready (kiki-runtime.target reached): <15s.
- Full boot to graphical login: <30s.

## Acceptance criteria

- [ ] systemd 257+ is the init.
- [ ] All Kiki services have unit files following the conventions.
- [ ] Quadlet generates app units correctly.
- [ ] sd_notify is used for readiness.
- [ ] Watchdog timeouts trigger restart.
- [ ] User session via systemd-user starts cage and agentui.
- [ ] systemctl, journalctl work for diagnostics.
